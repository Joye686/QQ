## 一、背景与目标

### 1.1 现状

当前成本账单流转链路：

```
业务上传账单 → ERP确认 → 推送付款申请 → 财务付款 → 金蝶记账
                                             （借：应付账款 贷：银行存款）
```

**缺失环节**：ERP 成本端（暂估）→ 金蝶（暂估入账、冲暂估转应付）无自动推送。

### 1.2 目标

打通 ERP 成本暂估模块与金蝶的凭证推送链路，实现：

1. **月末自动暂估入账**：ERP 成本模块暂估当月成本后，自动推送金蝶生成暂估凭证
2. **账单确认时冲暂估转应付**：业务在 ERP 确认账单时，自动冲销对应暂估并转正式应付
3. **跨月差异自动调整**：账单月份与暂估月份不一致时，差额自动计入当期成本
4. **全程可追溯**：暂估单、账单、金蝶凭证三向关联，支持对账和审计

## 二、业务流程

### 2.1 整体流程图

```
┌─────────────────────────────────────────────────────────────┐
│                      月末暂估流程                            │
│  ERP成本模块 ──暂估汇总──→ ERP暂估单 ──推送──→ 金蝶暂估凭证   │
│  (每月末自动触发)          (按供应商)         (借:成本 贷:暂估) │
└─────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────┐
│                   账单确认冲暂估流程                         │
│  业务确认账单 ──匹配暂估──→ ERP冲暂估单 ──推送──→ 金蝶冲销   │
│  (按供应商FIFO)     (红字冲暂估)        凭证+转应付凭证       │
└─────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────┐
│                      付款流程（已有）                        │
│  财务付款 ──推送──→ 金蝶付款凭证                            │
│              (借:应付账款 贷:银行存款)                        │
└─────────────────────────────────────────────────────────────┘
```

### 2.2 月末暂估流程

| 步骤 | 触发条件 | 操作 | 输出 |
|------|----------|------|------|
| 1 | 每月末自动触发 | ERP 成本模块汇总当月全部成本 | 成本汇总表 |
| 2 | 步骤1完成后 | 按供应商维度生成暂估单 | 暂估单（含暂估单号、供应商、金额、月份） |
| 3 | 暂估单生成后 | 推送金蝶生成暂估凭证 | 金蝶凭证号回填 |

**会计分录**：

```
借：成本类科目（按成本类型映射）
  贷：应付账款-暂估应付（按供应商辅助核算）
```

### 2.3 冲暂估转应付流程

| 步骤 | 触发条件 | 操作 | 输出 |
|------|----------|------|------|
| 1 | 业务在 ERP 确认账单 | 按供应商匹配暂估单池（FIFO） | 匹配结果 |
| 2 | 匹配完成后 | 生成冲暂估单 + 转应付单 | 冲暂估单、转应付单 |
| 3 | 单据生成后 | 推送金蝶（红字冲暂估 + 蓝字转应付） | 金蝶凭证号回填 |

**匹配策略**：先进先出（FIFO），按供应商维度匹配。

**会计分录**：

```
① 冲暂估（红字或反向分录）：
  借：成本类科目        -X（红字）
  贷：应付账款-暂估应付  -X（红字）

② 转正式应付：
  借：成本类科目         X
  贷：应付账款-应付账款   X
```

**差额处理**：账单金额 > 暂估金额时，差额自动计入当期成本，无需单独分录。

### 2.4 付款流程（已有链路，保持不变）

```
借：应付账款-应付账款
  贷：银行存款
```

## 三、数据模型

### 3.1 暂估单表（erp_estimate）

| 字段名 | 类型 | 说明 |
|--------|------|------|
| estimate_no | varchar(32) | 暂估单号（主键），格式：ZG+YYYYMM+序号 |
| supplier_code | varchar(32) | 供应商编码 |
| supplier_name | varchar(128) | 供应商名称 |
| estimate_month | varchar(7) | 暂估归属月份，格式：YYYY-MM |
| estimate_amount | decimal(18,2) | 暂估金额 |
| reversed_amount | decimal(18,2) | 已冲销金额，默认0 |
| status | varchar(16) | 状态：ESTIMATING（暂估中）/ PARTIAL（部分冲销）/ REVERSED（已冲销） |
| related_bill_no | varchar(64) | 关联账单号（冲销后回填，多个用逗号分隔） |
| kingdee_voucher_no | varchar(64) | 金蝶凭证号（推送后回填） |
| created_time | datetime | 创建时间 |
| updated_time | datetime | 更新时间 |

### 3.2 冲暂估记录表（erp_reverse_record）

| 字段名 | 类型 | 说明 |
|--------|------|------|
| record_id | bigint | 主键 |
| estimate_no | varchar(32) | 暂估单号 |
| bill_no | varchar(32) | 账单号 |
| reverse_amount | decimal(18,2) | 本次冲销金额 |
| diff_amount | decimal(18,2) | 差额（账单-暂估，正数为补差，负数为结余） |
| kingdee_voucher_no | varchar(64) | 金蝶凭证号 |
| created_time | datetime | 创建时间 |

### 3.3 推送日志表（erp_push_log）

| 字段名 | 类型 | 说明 |
|--------|------|------|
| log_id | bigint | 主键 |
| push_type | varchar(16) | 推送类型：ESTIMATE（暂估）/ REVERSE（冲暂估）/ PAYABLE（转应付）/ PAYMENT（付款） |
| source_no | varchar(32) | 源单号（暂估单号或账单号） |
| request_body | text | 推送请求体（JSON） |
| response_body | text | 金蝶响应体（JSON） |
| status | varchar(16) | 状态：SUCCESS / FAILED / RETRYING |
| retry_count | int | 重试次数，默认0 |
| created_time | datetime | 创建时间 |

## 四、接口设计

### 4.1 ERP → 金蝶推送接口

**接口地址**：`POST /api/kingdee/voucher/push`

**请求体**：

```json
{
  "voucherType": "ESTIMATE",
  "supplierCode": "SUP001",
  "supplierName": "XX供应商",
  "estimateNo": "ZG202604-001",
  "billNo": "",
  "costMonth": "2026-04",
  "amount": 100000.00,
  "debitAccount": "5001.01",
  "creditAccount": "2202.01",
  "remark": "4月成本暂估"
}
```

**字段说明**：

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| voucherType | string | 是 | 凭证类型：ESTIMATE（暂估）/ REVERSE（冲暂估）/ PAYABLE（转应付） |
| supplierCode | string | 是 | 供应商编码 |
| supplierName | string | 否 | 供应商名称 |
| estimateNo | string | 冲暂估时必填 | 暂估单号 |
| billNo | string | 转应付时必填 | 账单号 |
| costMonth | string | 是 | 成本归属月份 YYYY-MM |
| amount | decimal | 是 | 金额（冲暂估时传负数） |
| debitAccount | string | 是 | 借方科目编码 |
| creditAccount | string | 是 | 贷方科目编码 |
| remark | string | 否 | 摘要备注 |

**响应体**：

```json
{
  "success": true,
  "voucherNo": "KP-202604-00123",
  "message": "凭证生成成功"
}
```

### 4.2 幂等性保证

- 接口需支持幂等，通过 `estimateNo + billNo + voucherType` 组合判断是否已推送
- 重复推送时返回已有凭证号，不重复生成

## 五、匹配策略详解

### 5.1 FIFO 匹配规则

按供应商维度，从暂估单池中按**暂估月份从早到晚**顺序匹配：

```
供应商A的暂估单池：
  ZG202601-001  100,000  状态:ESTIMATING
  ZG202602-001  150,000  状态:ESTIMATING
  ZG202603-001  120,000  状态:ESTIMATING

场景1：收到账单 250,000
  → 冲 ZG202601-001（100,000）→ 状态:REVERSED
  → 冲 ZG202602-001（150,000）→ 状态:REVERSED
  → 差额: 0

场景2：收到账单 80,000
  → 冲 ZG202601-001（80,000）→ 状态:PARTIAL，剩余20,000

场景3：收到账单 350,000
  → 冲 ZG202601-001（100,000）→ 状态:REVERSED
  → 冲 ZG202602-001（150,000）→ 状态:REVERSED
  → 冲 ZG202603-001（120,000）→ 状态:REVERSED
  → 差额: -20,000（补差计入当期成本）
```

### 5.2 部分冲销处理

- 暂估单部分冲销后，剩余金额继续留在池中
- 状态标记为 `PARTIAL`，可继续被后续账单匹配
- 累计冲销金额 = 暂估金额时，状态更新为 `REVERSED`

## 六、异常处理

### 6.1 推送失败

| 场景 | 处理方式 |
|------|----------|
| 网络超时 | 自动重试3次，间隔5分钟 |
| 金蝶返回错误 | 记录错误信息，标记为FAILED，人工介入 |
| 科目映射缺失 | 推送前校验，提前拦截并告警 |

### 6.2 数据一致性

| 场景 | 处理方式 |
|------|----------|
| 暂估单重复生成 | 按 `供应商+月份` 做唯一约束 |
| 账单重复确认 | 推送前检查是否已生成冲暂估单 |
| 金额不一致 | 记录差异日志，触发人工审核流程 |

### 6.3 预警机制

| 预警条件 | 预警方式 |
|----------|----------|
| 暂估单超过3个月未冲销 | 邮件/消息通知业务负责人 |
| 推送失败超过重试次数 | 邮件通知IT运维 |
| 差额超过暂估金额20% | 邮件通知财务负责人 |

## 七、非功能性需求

### 7.1 性能

- 月末暂估推送：支持批量处理，1000条暂估单推送时间 < 5分钟
- 账单确认冲暂估：实时推送，响应时间 < 3秒

### 7.2 安全

- 接口传输使用 HTTPS + Token 认证
- 敏感字段（金额、供应商）加密存储

### 7.3 可追溯性

- 所有推送操作记录日志，保留至少3年
- 支持按暂估单号、账单号、凭证号双向查询
